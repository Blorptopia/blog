Behind tross

A quick refresher on CSVs

A CSV file is a simple tabular data format on the surface, where columns are delimited by commas, and the first row is usually the names of the columns.

As with everything there's usually a few different alternative ways to write CSVs, for example by excluding the header row and instead requiring the user of the data to pass in the names of the columns themselves.

What does tross do?

At its core tross is a library to serialize and de-serialize CSV files. Simply put, tross has two jobs:

  1. Converting the raw CSV contents into a map of strings
  2. Converting objects into a CSV

Tross does not handle the reading and writing of the file however it does handle streaming parsing and writing.

Tross has both a multi threaded and a single threaded variant of the serializer and de-serializer. This split was made due to the multi threaded being faster overall, but requiring quite a high up front cost to initialize all the web workers needed to parallelize the work. This means that it is preferred to use a single threaded parser for small files, and a multi threaded one for larger ones if you want the most performance.

The serializer

Single threaded

The single threaded serializer just requires escaping each column and joining them by the column separator. No fancy magic is done here.

Multi threaded

The multi threaded variant uses a slight variant of a worker pool to distribute tasks to a pool of workers. Since this is targeting JS, we use the web-worker wrapper around web workers and node's worker_threads. The slight variant of this pool being that it is pull-based instead of push based. In practice this means that the main thread defines a poll function that is called by the worker pool whenever a worker is available to handle work. This allows us to easily control the order tasks are executed in at the cost of the workers going idle between tasks while waiting for the main thread to send it a task

This is essentially an AsyncIterator, however with a few important changes:

  1. AsyncIterator provides a value, then waits for the consumer to consume it. This isn't a big issue in the writer, however it becomes a problem later with the parser.
  2. AsyncIterator doesn't fit well with tross requesting values one at a time from various different functions

While yes you should generally follow standards, the benefits of not following it here outweigh the benefits of following it.

The main thread is not only responsible for dispatching the tasks - it is also responsible for yielding the results back to the user and the serialization of the initial header row.

Potential optimizations

Sharing memory across threads

When sending messages to a web worker using postMessage, the data you send is usually cloned, however this can be changed by sending transferable objects and marking them as being transferred in the call to postMessage. The issue with this optimization however is that it does not support sending strings, only ByteArrays and similar types. I would assume that encoding and decoding the text you send to workers would have a higher overhead than cloning them, but I have not tested this.

Opt out of consistent ordering

We could have added an option to disable the guarantee that the order of serialized rows matches the order of rows given as input, as this would allow us to give you the result of tasks in any order instead of waiting for the tasks to complete in order before yielding them to the user. This may give you the results slightly earlier as we wouldn't have to wait for other tasks to complete, however this wouldn't be a big speedup due to the worker pool prioritizing putting tasks in order.

What is missing

The writer does not emit the non-standard separator header. There isn't any interesting technical challenge behind this choice, it's simply just not implemented as none of the data that I've needed to process has used this.

The deserializer

Single threaded

The single threaded variant works by going character by character and switching between different modes, switching modes when it reaches a control sequence

and in the case of control sequences it will peek forward and process the entire control sequence in one iteration.

Multi threaded

Now, multi threading is where the most interesting parts come in. The first thing that needs doing on the main thread is attempting to split the input buffer into roughly evenly sized chunks starting and ending at the exact start and end of a row. We do this by just seeking ahead a bit, find the next newline and use that as a chunk boundary. This isn't perfect, however I will come back to how we handle this later.

Once we have created the chunks, we create a task for every one of the chunks. This task includes a persistent identifier, as well as the range (not the actual content) of the chunk.

We then use the same worker pool mentioned while describing the multi threaded serializer to distribute tasks, prioritizing tasks from the start both to avoid

The parsing code running on the worker is quite similar to the one running using single threaded mode with three exceptions:

  1. Rows are not yielded one at a time, rather the entire task is finished before sending the result to the main thread.
  2. It operates on the entire input buffer but keeps to the range it was told to.
  3. The parser will attempt to stop at the end of the range it was given, however if that isn't at the end of a row it will continue going until the end of the row.

Once the result arrives at the main thread, the main thread links the result to the task. If the task exceeded it's designated range, it will replace the task that comes after with a new task that starts after the actual range the task that completed used.

After that it goes in order from the first task to the last task, yielding rows to the user until it encounters a task which isn't complete yet.

What is missing?

The writer does not emit the non-standard )separator header. There isn't any interesting technical challenge behind adding this, it's simply just not implemented. The current behavior is that it is treated as any other row, which usually leads to errors due to a mismatch between column counts or columns not being found due to sep=, being the only column name.

Potential optimizations

Lazily create tasks

Currently tasks are created when ingesting a chunk of the input buffer, however we could perhaps lazily create chunks when the poll function is created to avoid all that initial up front work, and in some cases allow of to avoid the mini chunks created at the end of an input chunk.

Preserve tasks that are thrown away

Sometimes the results of tasks that have completed is thrown away due to a previous task exceeding it's bounds and going into another task, causing the task and its result to be thrown away.

Lets imagine we have a series of events for three tasks:

  1. The third task completes first, fitting the bounds requested.
  2. The second task completes second, exceeding its bounds into task three causing the result of task three to get discarded and a new task three getting created to fit the new bounds.
  3. The first tasks completes exceeding its bounds into task two, causing the result of task two to get discarded and a new task two to getting created to fit the new bounds.
  4. Task two now completes again, this time without exceeding its bounds.

We now currently compute the result of parsing from the end of task two to the end of the buffer twice considering the result is fully thrown away when a task is rescheduled. If we stored the results of tasks that we threw away somewhere, we might be able to re-use them.

Streaming the input buffer on request

Currently the entire input buffer is sent to the worker to allow it to exceed it's bounds and continue going until it reaches the end of a row. This requires cloning a large string, which isn't ideal both for memory usage and for performance.

Instead of sending the entire input buffer, we could just send the "optimal" buffer initially but allow the worker to request additional chunks as needed. This might be slower however, I haven't benchmarked it yet.
